\section{Application Overview}

\textbf{ChefInPocket} is a mobile cooking-assistant application built with Flutter that guides users from ingredient selection through step-by-step cooking, with AI support at every stage. In this step, the wireframes from Step~2 have been transformed into \textbf{20 fully functional Flutter screens}, connected by a named-route navigation system, backed by a live REST API, and polished with custom typography and a consistent warm-neutral design language.

\subsection*{Core Features Implemented}
\begin{itemize}
  \item \textbf{Recipe Discovery} — Browse by cuisine (6 categories with emoji flags) or search by name with live autocomplete overlay.
  \item \textbf{Ingredient-Based Matching} — Multi-select ingredient picker calls the API and returns matching recipes instantly.
  \item \textbf{Serving Scale} — Stepper widget recalculates all ingredient quantities in real time via API.
  \item \textbf{AI Chef Chat} — Full-screen conversational assistant with recipe context badge, left/right chat bubbles, and auto-scroll.
  \item \textbf{Grocery List} — Item cards with dynamic deletion; quick-add bottom sheet with 12 ingredient chips.
  \item \textbf{Community \& Social} — Feed with Q\&A posting, bookmarking, and user-profile browsing.
  \item \textbf{Saved Recipes} — Bookmarked recipes stored via API, removable with a single tap.
  \item \textbf{My Recipes / Add Recipe} — Users can publish and view their own community posts.
  \item \textbf{Customize Ingredients} — Replace or remove elements without breaking recipe proportions.
  \item \textbf{Cooking Steps} — All steps shown as numbered cards; \emph{Mark as Done} dialog returns to Home.
\end{itemize}

\subsection*{Navigation Branches}

\begin{enumerate}
  \item \textbf{Access Branch:} Onboarding $\rightarrow$ Register \emph{or} Login $\rightarrow$ Home.
        \texttt{pushNamedAndRemoveUntil} clears the auth stack so users cannot navigate back after signing in.
  \item \textbf{Discovery Branch:} Home $\rightarrow$ Browse Cuisine $\rightarrow$ Ingredient Picker $\rightarrow$ Recipe Results $\rightarrow$ Recipe Detail.
  \item \textbf{Recipe Tool Branch:} Recipe Detail $\rightarrow$ Serving Scale / Customize Ingredients / Cooking Steps / AI Chat (all via \texttt{pushNamed} with the recipe slug as argument).
  \item \textbf{Social Branch:} Community $\rightarrow$ Ask Q\&A or User Profile; the Q\&A form posts to the community feed and pops back.
  \item \textbf{Profile Branch:} Profile $\rightarrow$ Saved Recipes or My Recipes; bottom navigation uses \texttt{pushReplacementNamed} to swap root tabs without stacking.
\end{enumerate}

\section{Technical Features}

\subsection{Named Routes}

All navigation uses Flutter's named-route system. The file \texttt{lib/routes/app\_routes.dart} defines a single \texttt{AppRoutes} class whose every member is a \texttt{static const String}. The \texttt{MaterialApp} in \texttt{lib/app.dart} maps all 20 constants to their screen widget constructors and sets \texttt{initialRoute: AppRoutes.onboarding}.

Three navigation call patterns are used:
\begin{itemize}
  \item \textbf{\texttt{pushNamed}} with an \texttt{arguments} object — used for detail screens that need context (e.g.\ recipe slug, cuisine name). Arguments are retrieved via \texttt{ModalRoute.of(context)?.settings.arguments}.
  \item \textbf{\texttt{pushReplacementNamed}} — used by the bottom navigation bar so tapping a tab replaces the current root screen without stacking it.
  \item \textbf{\texttt{pushNamedAndRemoveUntil(\ldots, (r) => false)}} — used after login, register, logout, and \emph{Mark as Done}, ensuring no orphaned screens linger in the back stack.
\end{itemize}

\subsection{Utility Classes}

Three files in \texttt{lib/utils/} centralise all static design tokens. Every class has a private constructor (\texttt{ClassName.\_()}) so it cannot be instantiated; all members are \texttt{static} or \texttt{static const}:

\begin{itemize}
  \item \textbf{\texttt{AppColors}} — Full colour palette: \texttt{primary} (warm orange-brown), \texttt{primarySoft} (tinted fill for selected states), \texttt{background} (cream \texttt{\#FAF7F2}), \texttt{border} (subtle warm grey), \texttt{textPrimary}, \texttt{textMuted}. Consumed as \texttt{AppColors.primary} throughout every screen and widget without any hardcoded hex literals.
  \item \textbf{\texttt{AppTextStyles}} — Six \texttt{static const TextStyle} values (\texttt{display}, \texttt{title}, \texttt{subtitle}, \texttt{body}, \texttt{caption}, \texttt{sectionLabel}) each binding to either the \emph{Inter} or \emph{Syne} font family defined in \texttt{pubspec.yaml}.
  \item \textbf{\texttt{AppSpacing}} — Five named double constants: \texttt{xs=4}, \texttt{sm=8}, \texttt{md=16}, \texttt{lg=24}, \texttt{xl=32}. Every \texttt{SizedBox}, \texttt{Padding}, and \texttt{EdgeInsets} in the project uses these constants instead of magic numbers.
\end{itemize}

\subsection{Asset and Network Images}

\textbf{Asset images} are stored in \texttt{assets/images/} and declared in \texttt{pubspec.yaml}:
\begin{itemize}
  \item \texttt{onboarding-meal.jpg} — food collage rendered on the Onboarding screen via \texttt{DecorationImage(image: AssetImage(\ldots))}.
  \item \texttt{recipe-hero.jpg} — fallback hero shown in Recipe Detail when a recipe has no remote URL.
  \item \texttt{community-bowl.jpg} — fallback cover for community post cards that return a broken URL.
\end{itemize}

\textbf{Network images} are loaded via \texttt{Image.network(url)} for recipe cover photos sourced from Unsplash (e.g.\ \texttt{https://images.unsplash.com/photo-1504674900247-\ldots}). Every \texttt{Image.network} call includes an \texttt{errorBuilder} callback that returns the corresponding asset image, preventing broken-image placeholders on network failures.

\subsection{Custom Fonts}

Two external font families are bundled in \texttt{assets/fonts/} and declared in \texttt{pubspec.yaml} under \texttt{flutter.fonts}:

\begin{itemize}
  \item \textbf{Inter} — \texttt{assets/fonts/Inter.ttf}. Applied to \texttt{AppTextStyles.display}, \texttt{body}, \texttt{subtitle}, \texttt{caption}, and \texttt{sectionLabel}. Provides a clean, highly-legible sans-serif for all body-level UI text.
  \item \textbf{Syne} — \texttt{assets/fonts/Syne.ttf}. Applied to \texttt{AppTextStyles.title}. Its wider geometric letterforms create visual hierarchy between section headings and body text.
\end{itemize}

Both fonts are only consumed through \texttt{AppTextStyles} constants, meaning the entire app's typography can be changed by editing a single file.

\subsection{Card-Based List with Dynamic Item Removal}

Two screens satisfy this requirement using model classes defined in \texttt{lib/models/app\_models.dart}:

\textbf{GroceryListScreen} (\texttt{lib/screens/grocery\_list\_screen.dart}):
\begin{itemize}
  \item Model: \texttt{GroceryItem} with fields \texttt{id}, \texttt{title}, \texttt{checked}.
  \item Each item is rendered as a \texttt{Card} containing the title and a trailing delete \texttt{IconButton}.
  \item Pressing delete calls \texttt{ApiService.instance.removeGroceryItem(item.id)}, then executes \texttt{setState(() \{ \_items.removeWhere((e) => e.id == item.id); \})} — the card disappears immediately with no full-page reload.
\end{itemize}

\textbf{SavedRecipesScreen} (\texttt{lib/screens/saved\_recipes\_screen.dart}):
\begin{itemize}
  \item Model: \texttt{SavedRecipe} with fields \texttt{recipeSlug}, \texttt{title}, \texttt{author}, \texttt{imageUrl}, \texttt{description}.
  \item Each entry is a \texttt{Card} showing a network cover image, author, title, and a bookmark \texttt{IconButton}.
  \item Tapping the icon calls \texttt{ApiService.instance.removeSavedRecipe(slug)} and splices the item out of \texttt{\_savedRecipes} with \texttt{setState}, updating the list in real time.
\end{itemize}

\subsection{Form Validation with AlertDialog}

\textbf{LoginScreen} (\texttt{lib/screens/login\_screen.dart}) uses a \texttt{Form} widget controlled by a \texttt{GlobalKey<FormState>}:

\begin{itemize}
  \item \textbf{Email field} — \texttt{TextFormField} with \texttt{validator}: fails if empty or lacks \texttt{@}. Error text: \emph{``Enter a valid email''} shown inline in red.
  \item \textbf{Password field} — \texttt{TextFormField} with \texttt{validator}: fails if empty or fewer than 6 characters. Error text: \emph{``Use at least 6 characters''}.
  \item On submit, \texttt{\_formKey.currentState!.validate()} runs first. If it returns \texttt{false}, both fields display their error labels and no API call is made.
  \item On a successful API login, \texttt{showSuccessDialog(context, title: 'Welcome Back', \ldots)} is called. It renders an \texttt{AlertDialog} with title, message, and a \emph{Continue} button that navigates to Home with \texttt{pushNamedAndRemoveUntil}.
\end{itemize}

\textbf{RegisterScreen} and \textbf{AddRecipeScreen} follow the same validation-and-dialog pattern with their respective fields.

\subsection{Responsiveness}

Every screen is wrapped by the shared \texttt{ChefPage} widget (\texttt{lib/widgets/common\_widgets.dart}), which applies a two-level strategy:

\begin{enumerate}
  \item \textbf{Maximum-width constraint:} A \texttt{ConstrainedBox(constraints: BoxConstraints(maxWidth: 620))} keeps content from stretching unreadably wide on tablets and landscape phones while remaining aligned to \texttt{Alignment.topCenter}.
  \item \textbf{Adaptive horizontal padding:} A \texttt{LayoutBuilder} checks \texttt{constraints.maxWidth}. For screens wider than 700 logical pixels the horizontal padding is \texttt{40.0}; otherwise it is \texttt{20.0}, giving a native feel on both phones and larger devices.
\end{enumerate}

\texttt{IngredientPickerScreen} adds a third level: its own inner \texttt{LayoutBuilder} switches the ingredient grid between \textbf{4 columns} (screens $>$ 420 px wide) and \textbf{2 columns} on narrow devices. \texttt{Wrap} widgets are used for tag chips on Recipe Detail and Browse Cuisine so they reflow naturally at any width without overflow.
